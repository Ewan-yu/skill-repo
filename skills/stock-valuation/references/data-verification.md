# 数据验证机制详解

> 基于实战经验总结的数据验证和质量控制机制

---

## 一、数据真实原则

### 1.1 核心原则

- **绝不编造数据**：所有数据必须来自权威数据源，严禁估算或猜测任何数值
- **数据优先级**：mx-data（结构化数据）> mx-search（资讯研报）> MCP工具（历史数据）> 询问用户
- **数据验证**：关键数据需交叉验证，确认准确性后再使用
- **明确标注**：数据来源和分析假设需清晰标注

### 1.2 禁止行为

- ❌ 禁止编造或估算股价、PE、PB等任何数值
- ❌ 禁止猜测大盘点位（必须通过mx-data或MCP查询）
- ❌ 禁止使用"约""大概"等模糊词修饰具体数值
- ❌ 禁止使用MCP历史估值数据作为当前股价（MCP的get_valuation_metrics可能返回过时数据）

---

## 二、数据验证机制

### 2.1 股价数据验证流程（必须执行）

1. **使用mx-data获取最新行情**：`mx-data "[股票代码] 最新行情"`
2. **记录当前日期和股价**，确认数据时效性
3. **如需MCP补充历史数据**，仅使用get_valuation_metrics的历史分位数
4. **交叉验证** - 对比mx-data和MCP的当前股价，偏差>5%时需调查原因
5. **在报告中明确标注数据来源和获取时间**

### 2.2 数据异常识别标准

| 异常类型 | 标准 | 处理方式 |
|---------|------|---------|
| **股价偏差** | > 5% | 需调查数据源差异 |
| **PE极端值** | >200 或 <0 | 需验证数据准确性 |
| **PB极端值** | >20 | 需验证数据准确性 |
| **数据源冲突** | 不同来源不一致 | 以最新权威数据源为准 |
| **数据时间戳** | > 1天 | 需重新获取最新数据 |

### 2.3 数据时效性要求

| 数据类型 | 时效要求 | 说明 |
|---------|---------|------|
| **股价、涨跌幅** | 必须使用当日数据 | 不能用历史数据 |
| **估值指标（PE/PB）** | 必须使用最新交易日数据 | TTM口径 |
| **财务数据** | 注明报告期（如2025年Q3） | 季报/年报 |
| **大盘指数** | 当日最新点位 | 不能用历史点位 |

## 三、工具使用策略

### 3.1 数据工具优先级

| 工具 | 类型 | 开发者 | 核心功能 | 使用场景 | 数据特点 |
|------|------|--------|---------|---------|---------|
| **mx-data** | **妙想Skill** | 东方财富 | 获取股票、板块、指数、基金、债券的行情、主力资金、估值等数据 | ①实时行情和历史股价<br>②财务数据<br>③估值指标<br>④股东和高管信息 | 结构化数据，最权威 |
| **mx-search** | **妙想Skill** | 东方财富 | 获取实时新闻、事件跟踪、舆情监控、行业动态等 | ①最新研报和机构观点<br>②行业趋势和景气度<br>③公司重大事件和公告 | 资讯研报，实时性强 |
| **MCP** | MCP服务 | - | 历史财务数据、估值指标历史趋势 | ①历史估值分位数<br>②多季度财务对比<br>③补充mx-data不足的数据 | 历史数据，可能过时 |

> **重要说明**：mx-data 和 mx-search 是**妙想Skills**（东方财富开发的独立技能），不是MCP服务。调用方式与MCP不同。

### 3.2 mx-data搜索关键词

| 数据类型 | 搜索关键词 | 何时使用 |
|---------|-----------|---------|
| **实时行情** | "[股票代码] 最新行情" | 必须第一步执行 |
| **大盘指数** | "上证指数 最新点位" | 第二步执行 |
| **财务数据** | "[股票代码] 财务数据" | 获取营收、利润、ROE等 |
| **估值指标** | "[股票代码] 估值指标 PE PB" | 获取最新PE/PB/PS |
| **公司信息** | "[股票代码] 公司资料" | 了解公司业务 |
| **股东高管** | "[股票代码] 股东结构" | 获取股东信息 |
| **资金流向** | "[股票代码] 资金流向" | 获取主力资金 |

### 3.3 mx-search搜索关键词

| 搜索类型 | 搜索关键词示例 | 用途 |
|---------|--------------|------|

| **行业趋势** | "[行业名称] 最新趋势 景气度" | 获取行业景气度 |

| **管理层访谈** | "[公司名] 管理层 访谈 战略" | 获取管理层观点 |

### 3.4 MCP工具使用限制

**MCP是补充数据源**：仅在mx-data无法提供数据时使用。

**禁止使用**：
- ❌ 禁止使用get_valuation_metrics的当前股价：该数据可能过时，仅用于历史分位数分析
- ❌ 禁止使用get_historical_price的最近数据作为当前价：历史数据有时间延迟

**必须验证**：
- ⚠️ 任何MCP获取的股价数据必须与mx-data验证

**MCP工具使用场景**：

| 工具 | 用途 | 限制 |
|------|------|------|
| `get_market_overview` | 获取主要指数实时行情 | 仅当mx-data不可用时 |
| `search_stock` | 搜索股票代码 | - |
| `get_realtime_quote` | 获取个股实时行情 | 必须与mx-data对比 |
| `get_income_statement` | 获取利润表 | 至少8个季度 |
| `get_balance_sheet` | 获取资产负债表 | 至少8个季度 |
| `get_cash_flow_statement` | 获取现金流量表 | - |
| `get_valuation_metrics` | 获取历史估值指标 | 仅用于历史分位数 |
| `get_historical_price` | 获取历史股价 | 仅用于历史趋势 |

## 四、数据验证检查清单

### 4.1 实时数据验证（必须首先完成）

- [ ] **已使用mx-data获取最新行情**：`mx-data "[股票代码] 最新行情"`
- [ ] **已记录数据获取时间**：确保数据是当日或最近交易日
- [ ] **股价数据时效性确认**：数据日期与报告日期相差<3天
- [ ] **已使用mx-data获取大盘指数**：`mx-data "上证指数 最新点位"`

### 4.2 数据一致性验证

- [ ] **股价数据交叉验证**：如使用MCP补充数据，已与mx-data对比验证
- [ ] **偏差检查**：不同来源股价偏差<5%，否则已调查原因
- [ ] **极端值检查**：PE<200，PB<20，否则已验证数据准确性
- [ ] **数据来源标注**：所有关键数据已注明来源（mx-data/mx-search/MCP）

### 4.3 禁止事项确认

- [ ] 无编造、估算、猜测的数据
- [ ] 未使用MCP的get_valuation_metrics返回的当前股价（可能过时）
- [ ] 未使用get_historical_price的最近数据作为当前价（有延迟）
- [ ] 数据时间戳清晰，所有数据都有明确的日期标注

### 4.4 报告数据标注要求

- [ ] 股价数据已标注：数据来源和获取日期
- [ ] 财务数据已标注：报告期（如2025年Q3）
- [ ] 估值指标已标注：计算口径（TTM/静态）
- [ ] 大盘指数已标注：当日收盘点位和日期

## 五、历史数据验证要点

### 5.1 必须分析的历史数据

**盈利能力分析**（至少3-5年）：
- 营业收入及增速
- 归母净利润及增速
- 毛利率、净利率变化
- EPS（每股收益）趋势

**财务健康度分析**：
- 资产负债率
- 流动比率、速动比率
- 经营现金流
- 应收账款占比

**估值指标历史**：
- PE、PB历史分位数
- PS历史分位数

**股东回报分析**：
- ROE（净资产收益率）
- 分红率、股息率

### 5.2 异常数据识别

| 异常类型 | 标准 | 处理方式 |
|---------|------|---------|
| **单季度业绩波动** | > 50% | 需要查找原因 |
| **PE极端值** | >100 或 <0 | 需要分析原因 |
| **PB极端值** | >10 | 需要分析原因 |
| **营收与利润背离** | 不一致 | 需要分析毛利率变化 |

### 5.3 数据交叉验证

- 资产负债表净资产 ≈ 历史累积
- 利润表净利润 ≈ 现金流量表经营净利润
- EPS × 股本 ≈ 净利润

## 六、常见数据问题处理

### 6.1 股价数据出现偏差或过时

**问题表现**：
- 不同数据源股价偏差 > 5%
- PE/PB数值异常（PE>200，PB>20）
- 数据时间戳 > 1天

**处理步骤**：
1. **立即重新获取**：使用mx-data获取最新行情
2. **交叉验证**：对比不同数据源的股价
3. **使用最新数据**：以时效性最好的数据源为准（mx-data > MCP）
4. **记录差异**：在报告中注明不同数据源的差异
5. **分析原因**：确认是数据延迟、数据源问题还是计算错误

### 6.2 历史案例警示

**中科曙光案例**：
- 报告曾使用MCP数据导致股价偏差74%（53.32元 vs 实际92.76元）
- **原因**：MCP的get_valuation_metrics返回了过时数据
- **教训**：当前股价必须使用mx-data获取，MCP仅用于历史分位数

### 6.3 搜索不到股票数据

**处理**：
1. 确认股票代码是否正确（6位数字）
2. 尝试用股票名称搜索
3. 检查是否为退市或新上市股票
4. 询问用户提供股票代码

### 6.4 数据不一致

**处理**：
1. 对比不同数据源
2. 查看数据时间戳
3. 优先使用最新官方数据
4. 在报告中注明差异

### 6.5 数据显示异常高估/低估

**处理**：
1. 检查是否是盈利低谷期/高峰期
2. 查看是否有特殊事件影响
3. 分析是否是概念炒作
4. 在报告中充分揭示风险

## 七、MCP的get_valuation_metrics能否用于获取当前股价？

### ❌ 禁止使用

MCP的get_valuation_metrics返回的股价数据可能过时。

### 正确用法

- **仅用于历史分位数分析**：PE/PB的历史最小值、25%分位、50%分位等
- **当前股价必须用mx-data**：`mx-data "[股票代码] 最新行情"`
- **如需验证**：可用get_realtime_quote交叉验证，但仍以mx-data为准

---

## 八、本章核心结论

1. **数据真实是底线**：绝不编造数据，所有数据必须来自权威数据源
2. **mx-data是首选**：实时行情必须用mx-data获取
3. **MCP是补充**：仅用于历史数据，不能用于当前股价
4. **交叉验证是必须**：不同数据源偏差>5%时需调查原因
5. **明确标注是责任**：所有数据必须注明来源和时间
6. **异常识别是能力**：能识别数据异常并正确处理
